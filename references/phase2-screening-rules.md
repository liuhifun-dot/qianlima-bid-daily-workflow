# Phase 2 · 标讯筛选与正文复判规则

## 适用范围

用于从千里马导出的 Excel 中筛选照明、路灯、亮化、太阳能路灯、光伏工程、充电桩安装等可能适合跟进的项目，并通过正文和附件证据复判。

目标不是“关键词命中最多”，而是模拟有投标经验的人判断：这个项目是否真的可能属于我方业务范围、是否值得业务人员跟进、是否存在必须人工确认的边界。

## 输入

- Phase 1 导出的千里马 Excel。
- 标题、地区、金额、招标类型、截止日期、链接、正文摘要、附件信息。
- VIP 正文读取结果。
- 附件预览、下载解析或 OCR 结果。

## 输出

最终业务复判 JSON。每条项目必须包含：

- `final_decision`: `keep` / `needs_review` / `reject`
- `final_grade`: A / B / C / D
- `body_read_ok`
- `attachment_required`
- `attachment_read_ok`
- `evidence`
- `reject_reason` 或 `review_reason`
- `business_scope`
- `risk_flags`
- `source_url`

## 核心原则

- 先排除明显不可做业务，再识别可能业务机会。
- 标题只能做初筛，不能替代正文和附件证据。
- 推荐必须有证据，不能因为标题里有一个关键词就推荐。
- 模糊项目不要强行推荐，也不要强行排除；证据不足时进入待人工复核。
- 筛选结论要方便业务人员理解，不要只输出分数。

## 决策等级

### keep

可作为推荐候选或重点跟进，但仍不是“确认投标”。

条件：

- 标题、正文或附件明确出现我方可做范围。
- 有施工、安装、改造、维护、供货安装、工程量、技术规范、清单等证据。
- 没有硬排除项。
- 如果正文或附件是必要证据，必须读取成功。

### needs_review

需要人工复核。

条件：

- 项目可能有关，但证据不完整。
- 正文不足，附件可能包含关键范围。
- 标题是综合改造、城中村、道路提升、学校医院维护、村道建设等边界项目。
- 室内小额灯具更换、医院/学校照明维护等需要业务判断是否值得跟进。
- `attachment_required=true` 且 `attachment_read_ok=false`。

### reject

排除。

条件：

- 命中硬排除。
- 正文或附件证明不是我方业务。
- 只是服务、咨询、评估、检测、租赁、审计、设计、纯设备采购。
- 只有弱关键词，没有施工/安装/维护/工程范围证据。

## 证据门槛

强制规则：

- `body_read_ok=false` 的项目不能 final_decision=`keep`。
- `attachment_required=true` 且 `attachment_read_ok=false` 的项目只能 final_decision=`needs_review`。
- `evidence` 不能为空。
- 标题级 reject 也必须把硬排除原因写入 `evidence`。
- 推荐候选必须说明命中的业务范围，例如“太阳能路灯安装”“LED照明更换”“分布式光伏施工”。

证据优先级：

1. 附件中的招标文件、技术规范、工程量清单。
2. 千里马正文中的采购内容、建设内容、服务范围。
3. 标题和摘要。
4. 千里马标签和订阅关键词。

不要把“其他信息”里的订阅全局关键词当作项目真实关键词。

## 硬排除规则

只要标题、正文或附件明确属于以下类别，直接排除：

- 选聘资产评估
- 值班用房租赁
- 检验、检测、监测
- 环境影响评价、环评
- 可研、可行性研究
- 审计、预算审核、结算审核
- 纯咨询设计
- 纯方案编制、规划编制
- 纯设备采购且无安装、施工、调试、维护范围
- 纯物业、保洁、安保、劳务派遣
- 纯道路硬化、路面拓宽、土建修缮且无照明/电气范围
- 纯租赁、公寓、房屋、场地使用权
- 纯软件平台、系统服务、运维平台，无硬件和施工范围
- 资产处置、拍卖、产权交易

例外：

- 如果正文或附件明确包含路灯、照明、亮化、太阳能路灯、光伏施工、充电桩安装等可做范围，不按标题硬排除，转入 `needs_review` 或 `keep`。

## 重点业务范围

优先关注：

- 路灯新建、改造、维护、养护。
- 太阳能路灯、智慧路灯。
- LED照明、室外照明、停车场照明、楼梯间照明更换。
- 亮化工程、景观照明、夜景照明。
- 分布式光伏、屋顶光伏、光伏改造、光伏施工。
- 充电桩安装、充电设施建设、外电接入，必须有施工或安装范围。
- 村庄提升、道路提升、城中村改造中明确包含照明、路灯或电气安装的部分。

## 边界项目处理

### 学校和医院

学校、医院项目不能因为单位类型直接排除。

处理：

- 室内灯具更换、小额 LED 改造：通常标为 B/C 级推荐候选或待人工复核，说明“需业务确认是否跟进”。
- 纯设备采购：排除。
- 含安装、调试、施工、维护：保留或待复核。

### 村道、道路、城中村、百千万工程

这类标题可能包含路灯机会，也可能只是道路硬化。

处理：

- 标题只写道路硬化、路面拓宽、排水、边坡、绿化：初步排除。
- 标题或正文有路灯、照明、亮化、太阳能路灯、电气安装：转入正文/附件复判。
- 附件未读但标题存在综合改造机会：`needs_review`。

### 光伏

光伏标题不能直接推荐。

处理：

- 分布式光伏、屋顶光伏、光伏施工、安装调试：可推荐候选。
- 值班用房租赁、屋面租赁、资产评估、可研服务：排除。
- “四可改造”“扶贫项目改造”等必须看正文或附件确认是否含施工。

### 充电桩

充电桩标题不能直接推荐。

处理：

- 充电桩安装、外电接入、基础施工、配套建设：可推荐候选。
- 纯充电桩设备、配件采购、合同结果：排除。
- 智能车库、综合系统中只提到充电桩：待人工复核或排除，取决于正文范围。

## 人工经验判断框架

像有投标经验的人一样判断时，按顺序问：

1. 这是招标机会还是结果公告/合同公告/中标通知？
2. 项目要买的是工程、施工、安装，还是服务、咨询、评估、租赁？
3. 是否有我方可做范围的明确词：路灯、照明、亮化、光伏施工、充电桩安装？
4. 金额是否值得跟进？
5. 地区是否在可服务范围内？
6. 截止时间是否还来得及？
7. 业主类型是否可靠？
8. 需要资质是否可能匹配？
9. 证据来自标题、正文还是附件？
10. 如果业务人员看到这个推荐，是否能理解为什么推荐？

如果第 3 点没有明确证据，不要直接推荐。

## 初筛流程

1. 读取 Excel。
2. 保留原始导出数据，不要删除。
3. 标准化字段：标题、类型、地区、金额、截止日期、链接、摘要。
4. 去重：同标题、同地区、同链接或同采购单位高度相似时合并。
5. 执行硬排除。
6. 对剩余项目按标题和摘要分类：
   - 明确相关
   - 模糊相关
   - 低相关
   - 排除
7. 对明确相关和模糊相关项目读取正文。
8. 对正文不足且有附件的项目读取附件。
9. 输出最终复判 JSON。

## 正文读取规则

必须读取正文的情况：

- 标题命中业务词但边界不清。
- 标题是综合改造、道路提升、学校医院改造、村庄提升。
- 光伏、充电桩、储能项目。
- 标题有附件提示，例如“订阅词在附件中”。

如果正文读取失败：

- 先判断是否登录失效。
- 如果登录失效，恢复登录后重试。
- 如果页面提示权限、验证码或安全验证，停止并标记 `needs_review`。
- 不要把正文失败的项目推荐为 keep。

## 附件读取规则

必须读附件的情况：

- 正文没有采购内容，但附件有“预览”或“下载”。
- 正文只写“详见附件”。
- 标题或摘要显示“订阅词在附件中”。
- 项目边界取决于工程量清单、技术规范书、采购需求书。

附件读取顺序：

1. 在线预览文本。
2. 下载 Word/PDF/Excel。
3. PDF 文本提取。
4. PDF canvas OCR。
5. 压缩包内文件解析。
6. 仍无法读取时，标记 `attachment_required=true` 和 `attachment_read_ok=false`。

附件未读成功时，最终只能是 needs_review，不能是 keep 或 reject。

没有布尔字段或手工备注例外。若人工已经真实打开正文或附件完成复核，必须把真实证据写入 evidence，并同步更新 body_read_ok 或 attachment_read_ok；否则仍保持 needs_review。

## 人工复核修正规则

人工复核不是校验绕过开关。不能只设置 manually_reviewed 或补几个审核字段。要改变最终结论，必须补录实际查看的正文/附件证据、来源、时间和复核人，并让证据状态与事实一致。
## 输出字段规范

每条最终结果至少包含：

```json
{
  "title": "",
  "source_url": "",
  "region": "",
  "amount_wan": null,
  "deadline": "",
  "notice_type": "",
  "final_decision": "keep | needs_review | reject",
  "final_grade": "A | B | C | D",
  "business_scope": "",
  "body_read_ok": false,
  "attachment_required": false,
  "attachment_read_ok": false,
  "evidence": [],
  "risk_flags": [],
  "reject_reason": "",
  "review_reason": ""
}
```

## 校验规则

校验必须拦截：

- `keep` 但 `body_read_ok=false`。
- `keep` 但 `evidence` 为空。
- `keep` 但 `attachment_required=true` 且 `attachment_read_ok=false`。
- `reject` 但 `evidence` 为空。
- `needs_review` 没有 `review_reason`。
- 推荐候选数量和 Excel/钉钉消息不一致。

## 推荐文案规则

钉钉和 Excel 中不要写“确认投标”。使用：

- 推荐候选
- 重点跟进
- 需业务确认
- 待人工复核

推荐候选必须写明：

- 项目名称
- 地区
- 金额
- 业务范围
- 推荐理由
- 风险或待确认点

示例：

`B级推荐候选：小额 LED 照明更换，正文显示含安装，需业务确认是否跟进。`

## 人工复盘规则

前期 Agent 筛选能力未完全稳定时，Excel 必须保留原始导出数据工作表，方便人工复盘漏网项目和误判项目。

人工发现误判后，必须沉淀到规则：

- 是硬排除词缺失。
- 是标题边界判断错误。
- 是正文证据读取失败。
- 是附件未读导致不确定。
- 是业务经验判断需要补充。

不要只改当天结果，要更新可复用规则。

## 项目本体与背景服务词

硬排除必须回答“本次采购的标的是什么”，不能只证明正文出现过某个词。

1. 优先读取标题、项目名称、项目类型、采购范围、招标内容和工作内容。
2. 标题明确写监理、检测、咨询、评估、租赁，或采购范围明确购买这些服务时，按硬排除处理。
3. 禁止投标条款中的“监理人”、监理单位名称、监理人发出的开工通知、资格和合同管理条款均属于背景，不定义项目本体。
4. 标题写施工、EPC、工程总承包、安装或建设，正文又有工程量清单、施工工期、安装、调试、并网或施工资质时，不得仅凭背景服务词排除。
5. 同时存在工程和服务内容但主次不清时进入待人工复核，不猜测。
