## A. 2A 初筛流水线（硬过滤 → 词 → 分）

### A1. 硬过滤（不过不进评分）

| 条件 | 当前值 | 来源 |
|:---|:---|:---|
| 省份 | 地区以 **「广东」** 开头 | `hard_filter.province` |
| 类型 | 二级类型 = **「招标公告」** | `hard_filter.bid_type` |
| 金额 | **5～350 万**（招标估价÷10000） | `amount_min_wan` / `amount_max_wan` |
| 标题去重 | 规范化标题去重（含公告后缀差异） | 脚本 `normalize_title_for_dedup` |

### A2. 标题排除（进 excluded，默认不进正文池）

| 类型 | 行为 | 配置 / 代码 |
|:---|:---|:---|
| **直接排除词** | 标题含任一词 → 排除 | `exclude_keywords`（约 170 个：医疗/食品/监理/设计/硬底化/纯 LED 屏采购/光伏板采购…） |
| **核心照明保护** | 标题已有路灯/亮化/照明等保护词时，**泛排除词不直接杀**，仍进评分/复判 | 脚本 `CORE_LIGHTING_PROTECTION` |
| **绝对服务排除** | 如「路灯巡检服务」「舞台灯光」等 → **强制排除**（覆盖保护） | `ABSOLUTE_SERVICE_EXCLUDE` |
| **纯显示屏标题** | LED 屏/护目灯且无路灯/照明/亮化 → 排除 | `DISPLAY_ONLY_TITLE` |
| **延迟排除词** | 医院/学校/绿化等 → **不直接杀**，标 `deferred_exclude_kw` 给正文 | `deferred_exclude_keywords`（5 个） |
| **行业 0 分** | 标题无核心/相关词 → 排除「行业不匹配」 | `score_industry==0` |

### A3. 评分（满分约 100，权重在代码里写死）

| 维度 | 上限 | 规则摘要 |
|:---|:---|:---|
| 金额 | 30 | 50–350 万 → 30；20–50 → 22；5–20 → 12；>500 → 20；… |
| 行业 | 25 | 核心词每个 +5（先核心）；无核心才用相关词每个 +3 |
| 地区 | 15 | 中山 15 / 深广 13 / 莞佛 12 / 珠惠 11 / 其它广东 10 / 默认 3 |
| 资质（标题推断） | 15 | 市政/路灯/亮化等 15；改造安装 10；设计监理咨询 8；默认 5 |
| 时间（截止） | 10 | ≥15 天 10；≥7 天 7；≥3 天 4；更短 1 |
| 黑名单位 | 5 | 暂无数据，默认 5 |

**后处理门槛（经验）：**

| 规则 | 效果 |
|:---|:---|
| 标题有明确照明保护词且总分 &lt;50 | **抬到 50**（保证进正文复判池） |
| 未命中**核心词**但总分 ≥70 | **压到 69**（相关词只能进「中等/复核」，不能直接「高分推荐」） |

### A4. 分档输出

| 档 | 总分 | 后续 |
|:---|:---|:---|
| **recommended 高分** | ≥70 | 进 2B/2C/2D 候选 |
| **considered 中等** | 50–69 | 同样拉正文（召回宽） |
| **low_score 低分** | &lt;50 | 2D 按「初筛评分不足」多 **reject** |
| **excluded** | — | 2D 记初筛排除 |

**关键词只做宽召回，不等于最终推荐**（YAML 文件头注释 + 复判脚本一致）。

### A5. 核心词 / 相关词（摘要）

- **核心（28）**：路灯、道路照明、户外照明、景观亮化、亮化工程、智慧路灯、太阳能路灯、灯杆、庭院灯、投光灯…  
- **相关（38）**：照明、灯光、亮化、分布式光伏、充电桩、市政道路、老旧小区改造、城市更新、灯具采购…

## B. 2D 业务复判（正文证据 → keep / NR / reject）

入口：`bid_business_rejudge` 调 `classify_business_opportunity`（`bid_business_rules`）。

### B1. 证据来源

| 源 | 用途 |
|:---|:---|
| 标题 + 预览 | 主标的、服务词、土建词 |
| VIP 正文（去页脚/相关商机） | 范围与排除上下文 |
| 附件文本（可读时） | 补强证据；扫描件 OCR weak 可能不够 |

### B2. 决策优先级（从前到后，命中即返回）

| 序 | 规则 | 结果 |
|:---|:---|:---|
| 1 | 初筛已 exclude / low_score 合同路径 | 多 **reject**（带初筛依据） |
| 2 | **道路土建主标的**（硬底化/拓宽/硬化…）且标题无明确照明 | **reject** |
| 3 | **硬服务**（监理/检测/设计/咨询/评估/测绘…） | **reject** |
| 4 | 服务语境 + 标题无照明证据 | **reject** |
| 5 | **纯光伏物资/组件采购**（无施工动作词） | **reject** |
| 6 | **纯 LED 显示屏** 且无照明/亮化/路灯范围 | **reject** |
| 7 | 显示屏 + 同时有照明范围 | **needs_review**（主次不清） |
| 8 | 产品排除词且无范围/产品/施工 | **reject**；有范围则记风险 |
| 9 | 非照明主标的（石材/铺装/桥梁加固…）且标题无照明 | **reject** |
| 10 | 改造/养护/补短板/道路等模糊主标 + 文中有照明 | **needs_review**（附带照明） |
| 11 | 通用「灯具」无户外语境 / 室内语境 | **needs_review** |
| 12 | 学校/医院 + 照明/灯具 + 维护更换 | **needs_review**（户内/户外边界） |
| 13 | 光伏维护类有施工证据 | **needs_review** |
| 14 | 维护对象是线路/变压器/配电 | **reject** |
| 15 | 明确 **施工/产品/范围** 命中 + 证据够 | 倾向 **keep**（A/B 级） |
| 16 | 证据不足 / 主次不清 | **needs_review** 或 **reject** |

（15–16 的细节以 `classify_business_opportunity` 后半段为准：要有 scope/products/construction 等可做证据，且不被前面硬排除。）

### B3. 推荐等级（与 decision 配合）

| 级 | 含义 | 常见 decision |
|:---|:---|:---|
| A/B | 业务上较清晰可跟 | keep |
| C | 边界/附带/证据不足 | needs_review |
| D | 明确不做 | reject |

### B4. 为什么 keep 少、NR 多（P1-03 根因画像）

| 因素 | 机制 |
|:---|:---|
| 初筛宽、复判严 | 相关词（道路改造/城市更新）易进 50–69，正文再判「附带照明」→ NR 或 reject |
| 服务/咨询/检测词库大 | 标题带「审计/咨询/检测」即使有「路灯」也易硬服务 reject |
| 土建主标的优先 | 硬底化/拓宽类即使清单有零星路灯也 reject |
| 无核心词封顶 69 | 冲不进「高分推荐」，多靠 considered 进复判 |
| 通用灯具 / 学校医院 | 一律 C 级人工，不自动 keep |
| 附件 OCR weak | 非必读文件弱文本 → 证据不足 → NR（不挡通道） |

**本轮正式样例：** raw≈95 → 复判 keep=2 / NR=3 / reject=78 —— 通道正常，偏严在规则层。

## C. 可调旋钮清单（改规则时优先动这些）

| 旋钮 | 位置 | 调松效果 | 风险 |
|:---|:---|:---|:---|
| `exclude_keywords` 删词/改短词 | YAML | 少误杀 | 脏项目进池 |
| `deferred_exclude_keywords` 增加 | YAML | 医院/学校类更晚判 | 正文池变大 |
| `amount_min/max_wan` | YAML | 金额带变宽 | 小单/超大单 |
| 核心词扩充 | YAML | 更多 ≥70 | 泛化误推 |
| 「无核心词压到 69」 | screening 脚本 | 相关词可进推荐 | 道路工程误推 |
| 「照明保护抬到 50」 | screening 脚本 | 弱匹配也进正文 | 噪声 |
| `SERVICE_EXCLUDE` / 硬服务 token | business_rules | 少杀「含服务词的工程」 | 真服务进 keep |
| `ROAD_CIVIL_DOMINANT` | business_rules | 土建+零星灯可能 keep | 土建大包误推 |
| 附带照明 → keep 条件 | business_rules | NR 变 keep | 主标的误判 |

**建议：** 先改 YAML + 用同日 raw 导出做 A/B 对照；改 `business_rules` 前用固定 10～20 条金标案例回归。

## D. 与「通道验收」的边界

| 可算通道 PASS | 不算规则 PASS |
|:---|:---|
| 能下到 Excel、正文/必读附件、出日报、归档发钉钉 | keep≥N 或业务满意率 |
| soft-continue / 额度 0 继续 | 推荐线索「够不够」 |
