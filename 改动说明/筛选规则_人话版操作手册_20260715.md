# 筛选规则 · 人话版操作手册（怎么看、怎么改）

> **写给：** 要自己调规则、但看技术文档费劲的人  
> **日期：** 2026-07-15  
> **一句话：** 电脑现在不是「像业务经理读标书」，而是「先按漏斗硬筛标题，再按词表做正文复判」。你要改的，主要是 **词表** 和 **几道门槛**。

---

## 0. 先建立正确预期

### 人怎么筛（你的思维）

1. 看标题：这是不是我们能干的照明/路灯/亮化/光伏施工？  
2. 看主标的：是施工/供货，还是纯设计、监理、检测、咨询？  
3. 看上下文：路灯是主体，还是道路硬化里附带几盏灯？  
4. 看金额、地区、工期，再决定跟不跟。  

### 电脑现在怎么筛（真实机制）

| 阶段 | 像人做什么 | 实际靠什么 |
|:---|:---|:---|
| 第一步 | 丢掉明显不相关的省份/类型/金额 | 硬条件数字 |
| 第二步 | 标题像「不做的行业/服务」直接扔掉 | **排除词表**（约 170 个） |
| 第三步 | 标题像「可能有关」捞进来打分 | **正选词**（核心 + 相关）+ 打分 |
| 第四步 | 打开 VIP 正文/附件再想一遍 | 又是一套 **范围词 / 服务词 / 土建词** |
| 第五步 | 输出：推荐 / 待复核 / 排除 | keep / needs_review / reject |

所以：

- **靠排除法** 在「第二步、第四步」很重 → 你会觉得「不像人」。  
- **联系上下文** 只在第四步有一点点（例如：硬底化 + 没写路灯 → 排除；学校+灯具 → 待复核），**还不是真正读懂标书**。  
- 你后面若把「自己筛一份标书的全过程」录下来，我们可以反推成更接近人的规则（那是下一代，不在当前 YAML 里）。

## 总流程：下载之后一共几步？

```text
千里马导出 Excel
    │
    ▼
【第 1 步】硬过滤：广东？招标公告？金额 5～350 万？
    │  不通过 → 直接扔掉（不进后面任何词表）
    ▼
【第 2 步】标题去重
    │
    ▼
【第 3 步】标题「一票否决」：命中排除词？
    │  是 → 多数直接排除（有「照明保护」例外）
    ▼
【第 4 步】标题「正选捞人」：有核心词/相关词吗？打分
    │  无行业词 → 排除「行业不匹配」
    │  有 → 算出总分，分成：高分 / 中等 / 低分
    ▼
【第 5 步】读 VIP 正文 + 附件（2B/2C）
    │  只对高分+中等（以及部分低分路径）拉正文
    ▼
【第 6 步】业务复判（像二次审核）
    │  服务/土建/纯屏/纯组件… → 排除
    │  边界不清 → 待复核
    │  正文证据够 → 推荐 keep
    ▼
日报 Excel + 钉钉（推荐 / 待复核 / 排除摘要）
```

下面逐步拆开：**规则是什么、词有哪些、你想改改哪里**。

# 第 1 步：硬过滤（还不看「照明」这些词）

**人话：** 先只留「广东 + 招标公告 + 金额差不多」的行。

| 规则 | 当前值 | 你想改时动哪里 |
|:---|:---|:---|
| 省份 | 地区以 **广东** 开头 | `rules_current.yaml` → `hard_filter.province` |
| 类型 | **招标公告** | `hard_filter.bid_type` |
| 金额下限 | **5 万** | `hard_filter.amount_min_wan` |
| 金额上限 | **350 万** | `hard_filter.amount_max_wan` |

**例子：**

- 「北京 + 路灯」→ 第 1 步就没了（不是广东）  
- 「广东 + 中标公告」→ 没了（不是招标公告）  
- 「广东 + 招标公告 + 2 万」→ 没了（金额太小）  

**文件：**  
`assets/rules_current.yaml`（生产包）  
= 测试机 `filter_rules/rules_current.yaml`

### 3.2 延迟排除词（标题有这些，先不杀，等正文）

| 词 | 意思 |
|:---|:---|
| 人行道及绿化工程 | 可能附带灯，也可能没有 |
| 绿化工程 | 同上 |
| 医院 | 可能是医院照明，也可能是医疗设备 |
| 学校 / 学校采购 | 可能是校园灯具，也可能是教学设备 |

→ 标上「延迟」，进后面复判再看。

### 3.3 重要例外：核心照明「保护」

若标题里已有 **路灯 / 照明 / 亮化 / 灯具 / 太阳能路灯…** 等保护词：

- 普通排除词 **不一定** 再一票否决（避免「某某医院路灯改造」被「医疗」误杀）  
- 但下面几类 **仍然强制排除**：  
  - 路灯巡检服务、光伏组件清洗、舞台灯光/舞台照明、某些检测校验服务  
  - 纯 LED 屏标题（没有路灯/照明/亮化）

**你怎么改第 3 步：**

| 你的诉求 | 改法 |
|:---|:---|
| 「某某词不该杀」 | 从 `exclude_keywords` **删掉** 或改成更长、更准的词 |
| 「某某词必须杀」 | 在 `exclude_keywords` **加上**（优先短词，覆盖变体） |
| 「医院相关先别杀，看正文」 | 放进 `deferred_exclude_keywords`，不要放直接排除 |
| 「设计服务也想接」 | 删掉「设计服务 / 工程设计 / 照明设计…」相关排除词，并同步检查第 6 步服务硬排除 |

文件：只改 YAML 即可（多数情况不用改 Python）。

# 第 4 步：正选词 + 打分（「像我们业务」的标题才往下走）

**人话：** 排除完以后，看标题像不像照明/路灯/亮化/相关工程；像就打分，分高的优先进正文。

## 4.1 正选词分两档

### 档 A · 核心正选（更像主营）— 每个 +5 分，行业分上限 25

这些是「标题里出现就比较像我们」：

| 类别 | 词 |
|:---|:---|
| 路灯道路 | 路灯、道路照明、智慧路灯、灯杆、太阳能路灯、LED路灯、LED太阳能路灯、隧道照明 |
| 户外/工程 | 户外照明、户外照明工程、照明工程、市政照明、智慧照明、建筑照明、户外建筑照明 |
| 亮化景观 | 景观亮化、亮化工程、园林景观亮化、夜景亮化 |
| 灯具产品名 | 庭院灯、景观灯、洗墙灯、线条灯、投光灯、草坪灯、点光源、地埋灯、埋地灯 |

**关键经验门槛（写在脚本里，不在 YAML）：**

- 标题 **只有相关词、没有核心词** → 总分最高压成 **69**（进「中等」，不能单靠「道路改造」就变「高分推荐」）  
- 标题有明确照明保护词但分太低 → 最低抬到 **50**（保证还能进正文复判）

## 4.2 其它打分（帮排序，不是一票否决）

| 项目 | 满分 | 人话 |
|:---|:---|:---|
| 金额 | 30 | 50～350 万最甜；太小/太大分低一点 |
| 地区 | 15 | 中山最高，深广莞佛…，其它广东约 10 |
| 资质（标题猜） | 15 | 标题像市政/路灯/亮化 → 高 |
| 投标截止时间 | 10 | 剩的天数越多分越高 |
| 黑名单位 | 5 | 暂时默认满分 |

## 4.3 分出来三档（进不进正文池）

| 总分 | 档名 | 人话 |
|:---|:---|:---|
| ≥70 | 高分 recommended | 「标题很像」，优先读正文 |
| 50～69 | 中等 considered | 「可能有关」，也读正文 |
| &lt;50 | 低分 low_score | 后面复判多半当排除 |
| 行业 0 分 | 排除 | 标题完全没有核心/相关词 |

**你怎么改第 4 步：**

| 诉求 | 改法 |
|:---|:---|
| 少漏掉真路灯 | 核心词加词；或放宽金额 |
| 少捞「道路改造」垃圾 | 从 **相关词** 删「道路改造/市政工程…」；或保持「无核心压 69」 |
| 想让「充电桩」当主营 | 把充电桩挪到 **核心词**，并检查第 6 步是否当产品排除 |
| 想改「70 分门槛」 | 要改 `bid_screening_*.py` 里的 `>= 70` / `50`（YAML 改不了） |

# 第 5 步：读正文和附件（不是「规则词」，是取证）

**人话：** 对高分+中等项目，打开千里马 VIP 页，抓正文；有附件就下下来读（PDF/Word/ZIP…）。

- 读成功 → 第 6 步有「证据」  
- 读失败 → 很多会变成 **待复核**（不敢自动推荐）  

一般 **不在这一步改业务规则**；通道问题另说。

---

# 第 6 步：业务复判（第二次审核，更接近「主标的是什么」）

**人话：** 带着标题 + 正文 + 附件，再问一遍：

1. 是不是 **设计/监理/检测/咨询** 服务？→ 排除  
2. 是不是 **道路硬化/拓宽** 主体，灯只是附带？→ 排除  
3. 是不是 **纯 LED 屏 / 纯光伏组件采购**？→ 排除  
4. 学校/医院灯具、通用「灯具」、改造提升里夹灯 → **待复核**（让人看）  
5. 正文里有清晰的 **路灯/亮化/户外灯具/光伏施工** 证据 → **推荐 keep**

### 6.1 复判结果三种

| 结果 | 钉钉/日报里像什么 | 你要不要改规则 |
|:---|:---|:---|
| **keep 推荐** | 「推荐候选」 | 太少 → 略放宽 3/4/6 步 |
| **needs_review 待复核** | 「待人工复核」 | 太多 → 把部分边界改成 keep 或 reject |
| **reject 排除** | 排除摘要 | 误杀 → 收紧排除词或复判服务词 |

### 6.2 复判里「像人」的一点上下文（仍是词驱动）

| 情况 | 电脑怎么判 |
|:---|:---|
| 标题硬底化/拓宽，且标题没路灯/亮化 | 直接排除（正文有零星「路灯」也不救） |
| 标题有监理/检测/设计/咨询等硬服务 | 直接排除 |
| 标题有路灯 + 医院/学校 + 更换 | **待复核**（怕是室内小额灯具） |
| 标题只有「灯具」，语境像教室/办公室 | **待复核** |
| 标题只有「灯具」，且有户外/道路/公园语境 | 可往推荐靠 |
| 正文有明确施工/产品/范围证据且无硬排除 | **keep** |

词表在脚本：`bid_business_rules_20260602_v02.py`（改这里要会一点代码，比改 YAML 重）。

### 6.3 为什么你会觉得「排除法不像人」

因为第 6 步仍然是：

```text
if 标题/正文 含某些词:
    排除 or 待复核
else if 含另一些词:
    推荐
```

**不是：**

```text
读完整招标范围 → 理解甲方买的是什么 → 判断我司能力与利润
```

要逼近人的思维，需要你后面提供的「我自己怎么筛一份标」案例，才能总结成新逻辑（例如：先定主标的句子，再定可做段落，而不是全篇扫排除词）。

   看：排除多少、高分/中等各多少。

5. **记下改了什么** → 写进 `改动说明.md` 或 lessons，避免下次忘。

你可以任选一份真实标书/导出行，按自己习惯记：

1. 第一眼标题你怎么判断（跟 / 不跟 / 存疑）？  
2. 你打开正文后看哪几段（招标范围、采购清单、资格）？  
3. 哪些句子让你决定「做」或「不做」？  
4. 金额、地区、竞争、工期你怎么权衡？  

发给我后，我可以帮你：

- 总结成 **人的决策树**（真正上下文逻辑）
