# 网站 UI 变更 — 应对方法（Playbook）

> **性质：流程与决策方法，不是自动改码程序。**  
> 当前**不要**指望完全无人值守「网站一改脚本自愈」。本文件规定：发现改版后，Agent/人应按何步骤修、验、交。  
> 与 `issue-log` 配套：先记 issue，再按本 playbook 改。

---

## 0. 现实边界（先想清楚）

| 能力 | 说明 |
|:---|:---|
| **现有优势** | Phase1 已是「按钮文案驱动」而非纯死坐标：生成/下载/继续导出/刷新；DOM 扫 `cu-btn` 等 |
| **仍会挂的点** | 文案改名、结构大改、弹窗文案变、URL 变、反爬、额度/登录 |
| **自动应对能做到的** | 失败时截图+结构化日志；多选择器候选；文案模糊匹配；fail-fast 带 reason |
| **不能假装全自动的** | 无验收的「乱点直到像成功」——会下错文件、误点支付/删除 |

**原则：定位优先语义（文案/role/结构），坐标只作最后手段；任何 UI 修复必须可验证目标达成。**

## 1. 发现网站问题时：先列清目标（对应你问的 a）

在 `issue-log/entries/ISSUE-....md` 写死 **目标清单**，例如 Phase1 导出：

| # | 目标（可判定真假） | 成功判据 |
|:---:|:---|:---|
| G1 | 进入标讯订阅页且已登录 | URL/标题含 subscribe；无登录门 |
| G2 | 日期为昨天～今天 | `currentFilterParams` 或页上日期文案匹配 |
| G3 | 打开导出配置并确认全部结果+Excel | 对话框可见对应文案 |
| G4 | 进入导出记录并锁定**本次**目标行 | 时间/条数与点击导出时刻接近 |
| G5 | 操作列最终出现「下载」并得到 xlsx | 文件存在、size>阈值、可 openpyxl 打开 |

**禁止的模糊目标：**「点一下试试」「看起来对了」。

改代码前先勾选：本次只修 G?，不顺手大重构。

## 2. 解决步骤建议顺序（对应 b）

### 2.1 固定现场（先证据，后动手）

1. **停住失败 run**，保留 `日志/` 与 checkpoint，不要立刻重跑冲掉现场。  
2. **截图**（全页 + 关键弹窗区域）→ 路径记入 issue。  
3. **读结构化日志**（export_record_rows、dialog text、URL）。  
4. 若 CDP 仍在：用 eval **导出当前可点控件列表**（文案、tag、粗略位置），不要上来按旧坐标点。

顺序建议：

```text
日志/reason → 截图理解布局 → DOM/文案探测 → 设计新定位 → 单步验证点击 → 再接状态机
```

**截图 vs 先点：**  
- **先截图 + 读日志/DOM**，再点。  
- 唯一例外：已知无害的关弹窗（「我知道了」）可先关再截。

### 2.2 定位策略（从稳到脆）

1. **可见文案精确匹配**（「下载」「生成」「继续导出」）  
2. **模糊/正则**（额度弹窗多种写法）  
3. **结构**：目标行 tr/list-item 内的 `.cu-btn`，而不是全页第一个按钮  
4. **属性**：data-* / aria-label（若站点有）  
5. **坐标 click**：仅当 1–4 失败，且截图标定后，并加「点击后断言」

### 2.3 「点对了」的确认（必做）

每次试验性点击后，必须检查 **目标表 G\* 是否推进**，例如：

| 动作 | 点击后必须看到 |
|:---|:---|
| 批量导出 | 出现导出配置对话框 |
| 对话框导出 | URL 或导航到导出记录，或出现新行 |
| 生成 | 弹窗或行状态变化（禁止重复生成） |
| 下载 | 文件系统出现新 xlsx，或 CDP download 事件 |

若点击后无上述变化 → **记失败**，回滚假设，禁止当作成功写入主逻辑。

### 2.4 把方法写回 issue

在 issue「解法」中写：

- 旧选择器/文案  
- 新选择器/文案  
- 断言用的判据  
- 风险（是否可能点到别的行）

## 3. 测试验证（对应 c）

| 轮次 | 做什么 | 通过标准 |
|:---:|:---|:---|
| **T1 单步** | 只跑改动的子流程（如仅导出记录循环） | G 目标命中；日志无异常分支 |
| **T2 集成** | Phase1 only 或 1→2a 短链 | 有 xlsx 或预期 fail-fast reason |
| **T3 回归** | 相关旧路径（resume、额度 0、正常额度） | 不破坏已 resolved issue |

测试日志放 `日志/`，issue 里只链路径。

---

## 4. 冗余清理（对应 d）

改 UI 定位后检查：

- [ ] 是否留下已失效的旧坐标/旧文案分支且永不走到  
- [ ] 是否重复的「再点一次」兼容代码可合并  
- [ ] 调试用 `print`/临时截图是否过多（保留关键失败截图即可）  
- [ ] 文档 SKILL / phase1 是否与代码矛盾  
- [ ] **不要删** issue-log 历史；可把 resolved 留作考古  

「冗 RIDES」按 **冗余代码/文档** 理解：能删的是死代码与矛盾说明，不是问题档案。

## 5. 最终验证与交付（对应 e）

### 5.1 第三次测试

T3 通过后：

1. 写 **短报告**（可放 `日志/UI修复_YYYYMMDD.md` 或更新 issue 第 6～7 节）  
2. **副本**：改过的主脚本复制一份带日期的备份，例如：  
   `scripts/_backup/bid_export_auto_v1.py.YYYYMMDD_ui`  
   （或你们既有备份目录；**勿覆盖唯一源**）  
3. **向用户说明**（固定话术要点）：  
   - 网站哪里变了（文案/结构/URL）  
   - 你做了什么决策（改定位 / 加候选 / fail-fast / 暂不自动）  
   - 验证做了几轮、结果  
   - 文件：主路径 vs 副本路径  
4. **批准门禁**（默认）：  
   - 主文件 = Skill 源码包 `scripts/` 中正在用的那份  
   - 副本仅备份，**不能**被 prepare_runtime 误拷为第二入口  
   - 用户说「作为主文件」后，才替换源码并同步 runtime；issue 状态 → `resolved`

### 5.2 交付检查表

- [ ] issue 状态已更新  
- [ ] INDEX.md 已更新  
- [ ] 主脚本路径明确  
- [ ] 副本路径明确  
- [ ] 用户已知情网站变更与决策  
- [ ] 无密钥写入 issue/报告

## 6. 「自动处理」还能做什么（仅规划，本次不改代码）

| 层级 | 想法 | 价值 | 风险 |
|:---|:---|:---|:---|
| L1 | 失败自动截图 + reason 枚举 + 写 issue 草稿 | 高 | 低 |
| L2 | 多文案同义词表（配置化，非写死一处） | 高 | 中（误匹配） |
| L3 | 点击后强制断言，失败重试有限次 | 高 | 中 |
| L4 | 视觉模型点按钮 | 实验 | 高（不稳、成本） |
| L5 | 完全自愈无人工 | 不推荐作为主路径 | 极高 |

**推荐路线：** 先 L1～L3 工程化；UI 大改仍走本 playbook + 人工确认目标。

## 7. 与现有目录分工

| 目录 | 用途 |
|:---|:---|
| `issue-log/` | 问题工单 + 本 playbook |
| `日志/` | 某次 run 的原始输出与报告 |
| `lessons/` | 提炼后的通用教训（可从 resolved issue 摘要迁入） |
| `scripts/` | 主代码；UI 修复最终落点 |

---

## 8. 一句话

网站改版时：**先记 issue 与可判定目标 → 截图/DOM 再点 → 点击后断言 → 两轮测 + 清理冗余 → 第三轮测后交报告/副本/请用户批主文件。**  
不是「自动乱点直到绿」。
